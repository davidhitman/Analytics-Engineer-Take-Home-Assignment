# Analytics-Engineer-Take-Home-Assignment

My assumptions about the source schema is based on the business context described—analytics for Irembo TeleClinic’s virtual consultations. The schema includes at least separate source tables for "consultations," "clinical outcomes," and "intake forms," as reflected in the case study tables and dbt project structure. Each consultation has a unique identifier, likely consultation_id, which acts as the primary key and is used to join across outcomes and intake-related tables. The "consultations" table contains key fields like created_at, started_at, and patient_id. These are expected to be in consistent ISO-format timestamps, with recent data potentially suffering from timezone inconsistencies ([UTC] or [UTC+2]) due to a platform release. The "clinical outcomes" table stores results of doctor workflows such as referral_issued (boolean), diagnosis, medication, and many more, and is keyed by consultation_id. Intake forms are available at least for April and have a patient-side referral_requested (boolean) field, which was introduced on April 3rd.

There is linkage across raw tables using consultation_id, allowing for event-level classification of each case ("doctor referral," "patient requested," etc.). Data from test and seed accounts is present, distinguished by patient_id patterns starting with 'TEST', and should be excluded from analysis. All lookup values for event or status types are standardized (likely codes or enums), not free text. All required fields for metrics (consultation_id, created_at, started_at, fields used in tests) are not null for real production data; nulls may indicate upstream ETL issues. Timestamps are intended to represent consistent real-world time in UTC, but recent data quality issues mean additional validation and correction are required. The referral rate metric is meant to capture clinical action; “both” (doctor and patient) should be tracked separately from “only patient requested.” Some denormalization or flattening occurs within staging models to handle analytic queries. The schema follows best practices for healthcare analytics reporting—clarity in joins, field naming, and test coverage for clinical reporting. Where the schema, join logic, or field meaning was unclear, I relied on context clues and industry norms for similar health analytics stacks. I used AI tools (specifically GitHub Copilot and ChatGPT) to help with drafting, proofing, and exploring gap-filling for both assumptions and code snippets; all logic, structure, and choices were reviewed/refined by myself.
